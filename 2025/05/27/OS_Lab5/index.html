实验目的：

  1. 了解文件系统的基本概念和作用。
  2. 了解普通磁盘的基本结构和读写方式。
  3. 了解实现设备驱动的方法。
  4. 掌握并实现文件系统服务的基本操作。
  5. 了解微内核的基本设计思想和结构。

1.文件系统概述

1.1文件系统的设计与实现

📕文件系统：（为了便于管理和访问存放在外部存储设备上的数据）在文件系统中，文件是数据存储和访问的基本单位。

  • 外部存储设备驱动：通过系统调用的方式陷入内核，对磁盘镜像进行读写操作。
  • 文件系统结构：fs 目录下存放的是文件系统服务程序的代码；而 user/lib 目录中的 file.c、fd.c、fsipc.c 等文件则存放了文件系统的用户库。文件系统服务进程和其他用户进程之间使用 IPC机制进行通信
  • 文件系统的用户接口：通过一个用户态的文件系统服务来实现

整体代码框架：

  • tools 目录中存放的是构建时辅助工具的代码。我们将在其中实现 fsformat 工具，并借助它来创建磁盘镜像。
  • fs 目录中存放的是文件系统服务进程的代码。fs.c 中实现文件系统的基本功能函数，ide.c 中通过系统调用与磁盘镜像进行交互，主干函数在 serv.c 中。通过 IPC 通信与用户进程 user/lib/fsipc.c 内的通信函数进行交互。
  • user/lib 目录下存放了用户程序的库函数。fsipc.c 实现了与文件系统服务进程的交互，file.c 实现了文件系统的用户接口，fd.c 中实现了文件描述符。

区分一个很混淆的概念：

dev_\* 是通用设备接口，代表“我不关心你是哪种设备”；
ide_\* 是具体的磁盘（IDE 硬盘）驱动函数，专门处理磁盘操作。

2.磁盘驱动

2.1 内存映射

把指导书里的内容转成一系列人话：

1. “设备是怎么和 CPU 通信的？”

“每一种外设都是通过读写设备上的寄存器来进行数据通信”

2. “外设寄存器是怎么找到的？”

“这些寄存器被映射到指定的物理地址空间”

  • 每个设备在内存里有一个“固定的地址范围”，叫内存映射 I/O。
    • 比如：串口 console 在 0x180003F8，IDE 控制器在 0x180001F0
  • 你只要对这个地址读写，相当于在操作这个设备！
3. “MIPS 架构用哪块地址空间来访问设备？”

“使用 kseg1 段… 不经过 MMU，也不用缓存”

2.2 IDE磁盘

这段话就讲了：

磁盘有两种寻址方法，CHS(柱面-磁头-扇区) 是物理结构的老方法，复杂且不便用；LBA 是线性编号方式，简单方便，实验里我们就用这个，最多能访问128GB，系统通过一组寄存器告诉设备哪个扇区要操作。

在我们的实验中，扇区编号有 28 位，因此最多可以寻址 2^28 个扇区，即 128 GB 的磁盘空间。

🧱 一、什么是文件控制块（struct File）？

它是操作系统用来描述和管理文件的关键数据结构，每个文件或目录都有一个对应的 File 结构体：

  • f_name：文件名（最多 128 字节）

  • f_type：文件类型（普通文件 or 目录）
  • f_direct[10]：10 个直接指针，指向磁盘上的数据块
  • f_indirect：1 个间接指针，指向磁盘上的数据块
  • f_dir：指向所属目录（只在内存中有效）
  • f_pad：结构体补齐字段，使多个 File 恰好填满一个磁盘块

3.3 块缓存

总结人话：

MOS 操作系统为了加快磁盘访问，在用户进程中划出一段虚拟地址空间（从 0x10000000 开始，总共 1GB），作为磁盘块的缓存区；磁盘块读入内存时就映射到这段空间里，以后访问就可以直接用内存访问，不用每次都去慢慢读磁盘。

感觉和tlb有点类似：都是缓存机制，都是用“空间换时间”，但块缓存是 OS 层面的磁盘优化，TLB 是 CPU 层面的地址映射优化。

4 文件系统的用户接口

4.1 文件描述符

💡1. 什么是文件描述符（fd）？
  • 当用户程序要打开一个文件时，操作系统不会直接把文件暴露给你，而是给你一个编号（整数），这就是fd（文件描述符）。
  • 你可以用这个 fd 来读取、写入、关闭文件，操作系统会帮你完成真实的文件操作。
  • 它就像你在图书馆借书，拿到一个凭条（fd），以后你就凭这个凭条去读书、还书。

4.2 文件系统服务

MOS 操作系统中的文件系统服务通过 IPC 的形式供其他进程调用，进行文件读写操作。

用户进程需要进行文件操作时，使用 ipc_send、ipc_recv 与 fs_serv 进行交互

fs_serv 的主函数先做初始化工作，建立一个“文件打开表” opentab，用来记录有哪些文件被哪些进程打开了。

接着它会去初始化整个文件系统，知道磁盘结构怎么组织的，哪些块是空的、哪些块已经写了内容。

所有文件系统都靠一个叫“超级块”的结构来记录整个磁盘的布局，fs_init 首先就是把超级块读进来了解磁盘的基本情况。

一切准备就绪后，它就“上线营业”啦，开始调用 serve() 持续循环地等待和处理用户程序的 IPC 请求。

💡 第一部分：文件系统支持的操作类型定义

📌 文件系统支持的操作都用这类枚举值 FSREQ_* 来标识。例如：

  • 当用户想打开文件时，发出的 IPC 请求类型是 FSREQ_OPEN；
  • 想删除文件时，发送 FSREQ_REMOVE 请求。
💬 第二部分：用户怎么请求文件操作？

“用户程序在发出文件系统操作请求时，将请求的内容放在对应的结构体中进行消息的传递。”

👉 意思是：用户程序并不是直接调用文件系统的内部函数，而是：

  1. 把请求内容（比如路径、打开模式、要删除哪个文件）填进一个结构体；
  2. 通过 IPC 把这个结构体发送给文件系统服务进程 fs_serv；
  3. fs_serv 根据你填的请求类型（比如是 FSREQ_REMOVE）和参数，来做相应操作。
🧠 第三部分：fs_serv 怎么处理这些请求？

“fs_serv 进程收到其他进程的 IPC 请求后，IPC 传递的消息包含了请求的类型和其他必要的参数，根据请求的类型执行相应的文件操作（文件的增、删、改、查等）。”

也就是说：

  • fs_serv 是一个“文件服务中心”，随时监听是否有 IPC 请求。
  • 它接到一个请求后，先看这个请求的类型（比如是 FSREQ_REMOVE）。
  • 然后去执行相应的文件操作逻辑（比如查找这个路径、从磁盘上删除文件结构等等）。
  • 最后通过 IPC 把结果（成功/失败）返回给请求它的用户进程。

Exercise 5.13 完成 user/lib/file.c 中的 remove 函数。

remove（user/lib/file.c）

  • 这是用户进程调用的接口函数，也就是用户代码里调用 remove(path) 就会执行这个函数。
  • 它实际上调用了 fsipc_remove，把删除请求转发给文件系统服务。

fsipc_remove（user/lib/fsipc.c）

  • 这是用户进程和文件系统服务之间的 IPC 通信层函数。
  • 它负责把删除请求（路径信息）封装成消息，通过 IPC 发送给文件系统服务进程。
  • 类似于“消息打包+发出”的角色。

serve_remove（fs/serv.c）

  • 这是文件系统服务进程接收到删除请求后的处理函数。
  • 它调用内核级的 file_remove 函数执行真正的删除操作。
  • 处理完后，把删除结果通过 IPC 发送回用户进程。

file_remove（fs/file.c）

  • 这是文件系统的底层具体实现函数，真正修改磁盘数据结构，释放磁盘块，更新目录等。
  • 是删除操作的核心实现，负责完成“删除文件”的实际工作。
🧠 总结一句话

file.c 发起请求 → fsipc.c 封装请求发 IPC → serv.c 主循环派发处理 → fs.c 执行逻辑 → ide.c 操作磁盘。
